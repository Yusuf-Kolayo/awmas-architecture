Tooling

Operations that keep a multi-stack workspace runnable.

Three Laravel services and a Flutter app in one workspace is convenient for coordination, but it introduces local development friction. This page documents the operational tooling that keeps the system runnable day-to-day.

One-command multi-service boot

A single root-level bash script starts all three Laravel services in the background, assigns distinct ports, binds the client backend to the local network so a physical device can reach it, and traps Ctrl+C to shut everything down cleanly. 

# serve-all.sh (sanitized excerpt)

echo "Starting all Laravel backends..."

# Operations backend (internal staff portal)
cd admin-backend
php artisan serve --port=8000 &
PID_ADMIN=$!
cd ..

# Mobile API backend (Flutter client)
cd client-backend
php artisan serve --host=0.0.0.0 --port=8001 &
PID_CLIENT=$!
cd ..

# Oversight backend (executive portal)
cd board-backend
php artisan serve --port=8008 &
PID_BOARD=$!
cd ..

# Graceful shutdown: pressing Ctrl+C kills every backend
cleanup() {
  kill $PID_ADMIN $PID_CLIENT $PID_BOARD 2>/dev/null
  exit
}
trap cleanup SIGINT
wait $PID_ADMIN $PID_CLIENT $PID_BOARD

Editor configuration for a monorepo

When a Dart project is nested inside a larger workspace, the Dart analyzer does not pick it up by default. The workspace settings explicitly point the analyzer to the Flutter project so IntelliSense, autocomplete, and the device selector stay enabled. 

// .vscode/settings.json (sanitized excerpt)
{
  "dart.projectSearchPaths": [
    "client-frontend"
  ],
  "dart.flutterHotReloadOnSave": "all"
}

Launch configurations make it explicit which directory Flutter should run from, and provide a fallback profile for iOS debugging when the Dart Development Service (DDS) cannot bind to a local port on a physical iPhone. 

// .vscode/launch.json (sanitized excerpt)
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Launch Flutter App",
      "request": "launch",
      "type": "dart",
      "program": "lib/main.dart",
      "cwd": "client-frontend"
    },
    {
      "name": "Launch Flutter App (No DDS - iOS Fallback)",
      "request": "launch",
      "type": "dart",
      "program": "lib/main.dart",
      "cwd": "client-frontend",
      "env": { "FLUTTER_NO_DDS": "1" }
    }
  ]
}

Automated deployment pipeline

Deployments are automated through GitHub Actions. Pushes to the main branch trigger a workflow that securely connects to the production VPS, pulls the latest code, and runs a predictable sequence of Laravel optimization commands to ensure zero downtime and fresh caches. 

# .github/workflows/deploy.yml (sanitized excerpt)

name: Deploy to VPS

on:
  push:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Deploy to Server
        uses: appleboy/ssh-action@v0.1.10
        with:
          host: ${{ secrets.HOST }}
          username: ${{ secrets.USERNAME }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /var/www/awmas
            
            # Pull latest code and force sync
            git fetch origin main
            git reset --hard origin/main
            
            # Install production dependencies
            composer install --optimize-autoloader --no-dev
            
            # Clear all caches before re-caching
            php artisan config:clear
            php artisan route:clear
            php artisan view:clear
            
            # Run migrations
            php artisan migrate --force
            
            # Re-cache for production performance
            php artisan config:cache
            php artisan route:cache
            php artisan view:cache
            
            echo "Deployment successful!"

Architectural outcomes

This tooling setup reduces friction and removes human error from both the daily development loop and the production release cycle.

Predictable deployments

GitHub Actions eliminates manual SSH sessions and forgotten cache-clearing commands. The production server state always matches the repository state.

Frictionless local boot

One script replaces three separate terminals and reduces the chance of leaving orphaned Laravel ports behind during development.

Real-device API testing

The mobile API is bound to the local network instead of just localhost, so a physical iPhone on the same Wi-Fi can reach the development backend natively.

Monorepo editor harmony

The editor stays useful even though the Flutter project is nested under a Laravel-heavy workspace, and iOS debugging has a fallback path that bypasses the DDS networking layer when needed.
