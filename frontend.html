Frontend

A Flutter client built for real mobile conditions.

The Flutter app is not a thin client. It manages tenant context locally, handles resilient network communication with the backend, and brokers a complete payment-and-receipt flow through a WebView. The patterns below are taken directly from the real codebase.

Architecture shape

The mobile codebase is organized around three responsibilities that the rest of the app depends on: a single bootstrap path, a Riverpod-managed state graph, and a unified API service. Screens consume the state and never speak to the network directly.

Bootstrap layer

Firebase initialization, push token acquisition with progressive backoff, and the global navigation key all live in main.dart and app_global.dart.

State layer

Riverpod providers hold the active connection, contact, client, bills, payments, account statements, broadcasts, and FCM token — typed, observable, and rebuild-scoped.

Network layer

A single AppApi base class owns headers and request shaping. A StateProvider holds the active AppApiService so the connection key can be swapped at runtime.

Resilient app bootstrap with FCM retry

The first thing the app does is make sure Firebase is ready and a push token can be acquired. iOS has a known timing problem: APNs registration can finish after the FCM token request fires, which returns null. The bootstrap path handles that with progressive backoff, and it never blocks the widget tree on success. 

// lib/main.dart (sanitized excerpt)

class _MyAppState extends ConsumerState<MyApp> {
  bool _fcmInitialized = false;

  @override
  void initState() {
    super.initState();
    WidgetsBinding.instance.addPostFrameCallback((_) => _initializeFcmToken());
  }

  Future<void> _initializeFcmToken() async {
    if (!kIsWeb && AppGlobal.firebaseApi != null && !_fcmInitialized) {
      _fcmInitialized = true;

      // Initial small delay so iOS APNs can register first
      await Future.delayed(const Duration(seconds: 2));
      var token = await AppGlobal.firebaseApi!.initNotifications();

      // Progressive backoff for slow iOS APNs registration
      if (token == null) {
        for (int attempt = 1; attempt <= 2; attempt++) {
          await Future.delayed(Duration(seconds: attempt * 3));
          token = await AppGlobal.firebaseApi!.initNotifications();
          if (token != null) break;
        }
      }

      if (token != null && mounted) {
        ref.read(fcmTokenProvider.notifier).state = token;
        fcmToken = token;
      }
    }
  }
}

The widget tree is built with ProviderScope regardless of FCM outcome. The push token is treated as a side-effect, not a hard requirement, so a denied permission or a slow APNs registration never blocks the splash screen.

Platform-aware FCM token retrieval

Android and iOS have entirely different failure modes for push tokens. Android fails when Google Play Services is stale, and the fix is to invalidate and re-request. iOS fails when the APNs token is not yet registered, and the fix is to wait and retry. The FirebaseApi knows both. 

// lib/APIs/firebase_api.dart (sanitized excerpt)

Future<String?> _getTokenForPlatform() async {
  if (Platform.isIOS) return _getTokenForIOS();
  if (Platform.isAndroid) return _getTokenForAndroid();
  return _firebaseMessaging.getToken();
}

Future<String?> _getTokenForIOS() async {
  // Wait for the APNs token before requesting the FCM token
  final apnsToken = await _firebaseMessaging.getAPNSToken();
  if (apnsToken == null) {
    await Future.delayed(const Duration(seconds: 2));
  }
  var fcmToken = await _firebaseMessaging.getToken();
  if (fcmToken != null) return fcmToken;

  // iOS-specific retry with longer APNs registration delay
  await Future.delayed(const Duration(seconds: 3));
  return _firebaseMessaging.getToken();
}

Future<String?> _getTokenForAndroid() async {
  var fcmToken = await _firebaseMessaging.getToken();
  if (fcmToken != null) return fcmToken;

  // Android fix: invalidate and recreate, in case Play Services is stale
  try {
    await _firebaseMessaging.deleteToken();
    await Future.delayed(const Duration(seconds: 1));
    return await _firebaseMessaging.getToken();
  } catch (_) {
    return null;
  }
}

The _getTokenForPlatform() dispatcher means the rest of the codebase never has to ask "what platform am I on" — it just calls initNotifications() and gets the right behavior automatically.

Riverpod state graph and provider composition

Every long-lived piece of state in the app is a Riverpod provider. The active connection, the signed-in contact, the list of bills, the FCM token, the list of saved cards — all of them are observable, testable, and rebuild-scoped. The widget tree reads them with ref.watch, and mutates them with ref.read(...notifier).state = .... 

// lib/APIs/firebase_api.dart + lib/APIs/app_api_service.dart (sanitized excerpt)

// The FCM token is just a string held in a provider
final fcmTokenProvider = StateProvider<String?>((ref) => null);

// The active API service is a StateProvider so the connection can be
// swapped at runtime without losing listeners
final appApiServiceProvider = StateProvider<AppApiService>(
  (ref) => AppApiService(),
);

// Higher-level providers track the user identity
final clientProvider  = StateProvider<Client>((ref) => throw UnimplementedError());
final contactProvider = StateProvider<Contact>((ref) => throw UnimplementedError());
final connectionProvider = StateProvider<Connection?>((ref) => null);

// Collection providers are notifiers that know how to merge
// initial loads with refresh loads ("replace" vs "append")
final billsProvider = BillsNotifier();
final paymentsProvider = PaymentsNotifier();

The collection providers implement a populateFromResponse(json, mode) method that knows how to handle both the initial load and a pull-to-refresh on a screen. Screens never have to ask the question "is this the first load or a refresh?".

Runtime tenant context through shared headers

The Flutter app serves eleven different businesses, and the user can switch between them at any time. The AppApi base class owns the active Connection and stamps it into every outgoing request, so individual calls never have to remember to attach the tenant key. 

// lib/APIs/app_api_service.dart (sanitized excerpt)

class AppApiService extends AppApi {
  // Set the active connection on the base class — from this point on,
  // every request through this service carries the tenant key.
  void setHeadersConnection(String connectionKey) {
    // The base AppApi exposes a mutable headers map; the connection key
    // is stamped into a custom header read by the backend router
    headers['X-Tenant-Connection'] = connectionKey;
  }
}
// lib/controllers/app_controller.dart (sanitized excerpt)

static Future<Map<String, dynamic>> signIn(
  WidgetRef ref,
  String phone,
  Connection connection,
) async {
  final appApiService = ref.read(appApiServiceProvider);
  final fcmToken      = ref.read(fcmTokenProvider);
  final deviceData    = await getDeviceInfo();

  // Tenant context is swapped BEFORE the request, then stamped into headers
  appApiService.setHeadersConnection(connection.key);

  final response = await appApiService.signIn({
    'phone':      phone,
    'fcm_token':  fcmToken,
    'device_info': deviceData,
    'connection': connection.key,
  });

  if (response['status'] == 'success') {
    // Persist the device registration so the next launch is pre-authenticated
    await AppApiService.registerClientOnDevice(response['data']['contact']['phone'], connection);
    return {'status': true, 'comment': response['comment']};
  }
  return {'status': false, 'comment': response['comment']};
}

The device's active tenant is also written to SharedPreferences on sign-in, and read back on cold start, so the app re-opens on the right business even after being killed.

Unified request and upload handling

Every screen-level API call goes through a single handleRequest method on AppApi. The method pre-checks connectivity, branches between a multipart upload and a JSON POST, and translates the response into a uniform shape that the rest of the codebase can rely on. 

// lib/APIs/app_api.dart (sanitized excerpt)

Future<Map<String, dynamic>> handleRequest({
  required String url,
  required Map<String, dynamic> body,
  File? file,
  String? fileField,
}) async {
  // 1. Pre-flight connectivity check
  final connectivity = await (Connectivity().checkConnectivity());
  if (connectivity == ConnectivityResult.none) {
    return {
      'status':  'failed',
      'comment': 'No internet connection detected on this device',
      'data':    <String, dynamic>{},
    };
  }

  // 2. Multipart upload path (e.g. payment confirmation receipt photo)
  if (file != null && fileField != null) {
    final request = http.MultipartRequest('POST', Uri.parse(url));
    headers.forEach((k, v) => request.headers[k] = v);

    // Stream from a real file (mobile) or raw bytes (web)
    if (body['is_web'] == true) {
      request.files.add(http.MultipartFile.fromBytes(
        fileField, file as Uint8List, filename: 'file.jpg',
      ));
    } else {
      final stream  = http.ByteStream(file.openRead());
      final length  = await file.length();
      request.files.add(http.MultipartFile(
        fileField, stream, length, filename: file.path.split('/').last,
      ));
    }

    final streamed = await request.send();
    final response = await http.Response.fromStream(streamed);
    return _parseResponse(response);
  }

  // 3. Standard JSON path
  final response = await http.post(
    Uri.parse(url),
    headers: headers,
    body: jsonEncode(body),
  );
  return _parseResponse(response);
}

The _parseResponse() helper centralizes the success/failure translation, so callers always get {status, comment, data} back — no matter which backend version produced it.

Binary PDF download with progress and fallback

Account statements and bills are downloaded as PDFs. The backend may stream the raw binary or return a JSON envelope with a base64 payload (the latter is the older deployment). The downloader is written to handle both transparently, with progress callbacks so the UI can show a real download bar. 

// lib/APIs/app_api.dart (sanitized excerpt)

Future<Map<String, dynamic>> downloadBillPdf(
  String billId, {
  void Function(int received, int? total)? onProgress,
}) async {
  final request = http.Request(
    'POST',
    Uri.parse('$apiBaseUrl/client/download_bill_document'),
  );
  request.headers.addAll({...headers, 'Accept': 'application/pdf,application/json'});
  request.body = jsonEncode({'bill_id': billId});

  final client   = http.Client();
  final streamed = await client.send(request);
  final ct       = streamed.headers['content-type'] ?? '';

  // Path 1: server streams raw PDF bytes — preferred
  if (streamed.statusCode == 200 && ct.contains('application/pdf')) {
    final total   = streamed.contentLength;
    int received = 0;
    final chunks  = <int>[];

    await for (final chunk in streamed.stream) {
      chunks.addAll(chunk);
      received += chunk.length;
      onProgress?.call(received, total);
    }

    return {
      'status':   'success',
      'mode':     'raw',
      'filename': _extractFilename(streamed.headers['content-disposition']),
      'bytes':    Uint8List.fromList(chunks),
    };
  }

  // Path 2: JSON fallback (base64 PDF or error envelope)
  final bodyString = await streamed.stream.bytesToString();
  return jsonDecode(bodyString) as Map<String, dynamic>;
}

The Content-Disposition header is parsed for the suggested filename, so iOS and Android can both offer a meaningful default when the user saves or shares the downloaded file.

WebView payment flow with deep-link reconciliation

Card payments are processed inside a hosted checkout page in a WebView. The Flutter side does not handle card details at all — it just opens the checkout URL, listens for the success callback URL, refreshes the user state, and navigates home. The user is never stuck on a payment screen. 

// lib/screens/payment/pay_final_screen.dart (sanitized excerpt)

final controller = WebViewController()
  ..setJavaScriptMode(JavaScriptMode.unrestricted)
  ..setNavigationDelegate(
    NavigationDelegate(
      onProgress: (int progress) {
        dev.log('progress: $progress');
      },
      onNavigationRequest: (NavigationRequest request) async {
        // The hosted checkout redirects here on success
        if (request.url.contains('/payment_successful')) {
          await AppController.refreshData(ref, context);
          Navigator.pushNamedAndRemoveUntil(
            context, HomeScreen.routeName, (route) => false,
          );
        } else {
          _showErrorDialog('Payment verification failed');
          Navigator.pop(context);
        }
        return NavigationDecision.navigate;
      },
    ),
  )
  ..loadRequest(Uri.parse(authorizedPaymentUrl));

AppController.refreshData() re-pulls the entire home state (bills, payments, account statements, broadcasts) so the user sees the new balance the moment the WebView resolves the success callback — no manual refresh required.

Platform-aware device fingerprinting

Every sign-in includes a structured device fingerprint. The shape of the fingerprint is different on Android, iOS, and Web, so the controller dispatches to a per-platform reader. The backend uses this for fraud signals, device tracking, and account recovery flows. 

// lib/controllers/app_controller.dart (sanitized excerpt)

static Future<Map<String, dynamic>> getDeviceInfo() async {
  final deviceInfo = DeviceInfoPlugin();
  try {
    if (kIsWeb) {
      return _readWebBrowserInfo(await deviceInfo.webBrowserInfo);
    }
    if (Platform.isAndroid) {
      return _readAndroidBuildData(await deviceInfo.androidInfo);
    }
    if (Platform.isIOS) {
      return _readIosDeviceInfo(await deviceInfo.iosInfo);
    }
  } catch (_) {
    return {"Error": "Failed to get device info"};
  }
  return {};
}

static Map<String, dynamic> _readIosDeviceInfo(IosDeviceInfo data) => {
  'name':               data.name,
  'systemName':         data.systemName,
  'systemVersion':      data.systemVersion,
  'model':              data.model,
  'isPhysicalDevice':   data.isPhysicalDevice,
  'identifierForVendor': data.identifierForVendor,
  // ...
};

The fingerprint travels with the FCM token and the tenant key as a single sign-in envelope, so the backend has all the context it needs to authorize the session in one round trip.

Thin controllers, fat services

AppController is a static class of method-level coordinators, not a stateful object. Each method reads the providers it needs, calls the API service, then writes the result back to the providers. Screens never call the network layer directly — they call a controller method and let the state graph react. 

// lib/controllers/app_controller.dart (sanitized excerpt)

static Future<Map<String, dynamic>> initializeHome({
  required WidgetRef ref,
  String contactIdSource = 'contact',
}) async {
  // Read everything we need from the state graph
  final appApiService = ref.read(appApiServiceProvider);
  final client        = ref.read(clientProvider);
  final contact       = ref.read(contactProvider);
  final connection    = ref.read(connectionProvider);
  final fcmToken      = ref.read(fcmTokenProvider);
  final deviceData    = await getDeviceInfo();

  // One network call
  final response = await appApiService.initializeHomeData({
    'contact_id':  contactIdSource == 'client' ? client.contactId : contact.id,
    'phone':       contact.phone,
    'property_id': client.propertyId,
    'connection':  connection?.key,
    'device_info': deviceData,
    'fcm_token':   fcmToken ?? '',
  });

  if (response['status'] != 'success') {
    return {'status': false, 'comment': response['comment']};
  }

  // Write the result back into the state graph in a single sweep
  final d = response['data'];
  ref.read(clientProvider.notifier).state           = Client.fromJson(d['client']);
  ref.read(contactProvider.notifier).state          = Contact.fromJson(d['contact']);
  if (d['bills'].isNotEmpty)             ref.read(billsProvider.notifier).populateFromResponse(d['bills'], 'replace');
  if (d['payments'].isNotEmpty)          ref.read(paymentsProvider.notifier).populateFromResponse(d['payments'], 'replace');
  if (d['account_statements'].isNotEmpty) ref.read(accountstatementsProvider.notifier).populateFromResponse(d['account_statements'], 'replace');
  if (d['message_broadcasts'].isNotEmpty) ref.read(messageBroadcastsProvider.notifier).populateFromResponse(d['message_broadcasts'], 'replace');

  return {'status': true, 'comment': response['comment']};
}

Screens that watch these providers rebuild automatically the moment the controller finishes writing. There is no manual setState() in the home flow — the state graph does the work.

Architectural outcomes

These patterns combine to create a mobile client that is responsive, recoverable, and trustworthy in real production conditions.

Resilient first-launch experience

iOS APNs registration delays and Android Play Services stalls are handled inside the FCM layer, so a slow push registration never blocks the splash screen or the sign-in flow.

Single source of truth in the state graph

Every screen reads from the same Riverpod providers. Refreshing the home state in one place propagates to every screen that watches those providers — no duplicated fetches, no stale data.

Forward-compatible network layer

The same download method handles streamed raw PDFs and JSON base64 envelopes. Older deployments keep working, newer deployments stream faster — the client never has to know which backend it is talking to.

Tenant context without boilerplate

The active business is stamped into request headers once, at the moment the connection is selected. Individual calls never have to remember to attach the tenant key, and switching tenants is a single header rewrite.
