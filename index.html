Case study

Three Laravel services, one Flutter client, one operational platform.

AWMAS is organized as a multi-application system with three distinct Laravel backends and one Flutter mobile client. Each application is responsible for a specific audience, a specific data boundary, and a specific set of workflows.

System shape

One workspace, four applications, three operational audiences.

The system was deliberately split into multiple deployable applications instead of handled as a single monolith. This creates stronger permission boundaries, clearer routing responsibilities, and easier reasoning about internal versus external workflows.

admin-backend

Operational dashboard for internal staff, billing workflows, tenant-aware administration, and data maintenance.

board-backend

Oversight surface for executives and stakeholders, focused on reporting, cross-business visibility, and governance.

client-backend + client-frontend

Secure mobile API and Flutter app pair for authentication, billing, payment flows, and document retrieval.

Request flow

How traffic moves through the platform.

The Flutter client talks to the client API. Internal users use the admin and board portals. Each application resolves its own tenant context from session state or request headers, and routes the request to the correct isolated database.

Mobile client → client backend

Authentication, billing, uploads, payments, PDF generation.

Operations staff → admin backend

Operational workflows with request-time database switching.

Executives → board backend

Reporting and oversight separated from operational CRUD.

Representative patterns

The logic that defines the system.

Tenant-aware database routing

Tenants are resolved dynamically. Internal users inherit context from session state. The mobile API resolves context from request headers and validated connection keys.

Service-layer domain rules

Billing and account logic live in dedicated services with fallback-safe financial mutations, normalized billing summaries, and document-ready payload shaping.

Deferred activity logging

Every staff action is captured in a request-scoped payload and written after the response is flushed, so audit overhead never blocks the user-facing request.

Process Authority approval workflow

Privileged actions are gated by an explicit permission slot. Junior requests are parked in an Admin Desk inbox for senior approval, all from the same controller path.

Immutable preserved billing

Every generated bill is snapshotted to a dedicated table with a content hash, so the printed document remains trustworthy even if the source data is later edited.
